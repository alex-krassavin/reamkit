# OOP / GRASP / паттерны — анализ reamkit (ветка v1)

> Только анализ. Кода нет — везде описаны **роли** и кто кому **делегирует**, словами.
> Каждое утверждение привязано к `файл:строка`. Метод: 11 модульных аналитиков +
> 3 сквозные линзы (GRASP-карта, карта паттернов, кодекс ограничений), затем
> адверсариальная верификация всех 41 предложения по трём кластерам — с проверкой
> «реален ли evidence», «честен ли byteRisk», «есть ли реальный второй потребитель».

## 0. Главный вывод (TL;DR)

Ты сказал: «паттерн делегирования прям напрашивается». После строгой проверки —
**да, но точечно, и это важно различать**:

1. **Делегирование реально напрашивается СЕЙЧАС ровно в 2 местах** — там, где уже
   есть второй живой потребитель роли (оба — `svg-writer`, второй адаптер PageDoc):
   - **проба/эмиссия изображений** (`embedImage` → роли `prepare` + `add`);
   - **порядок отрисовки `PageItem`** (общий «paint order» для PDF-emit и SVG).

2. **Ещё ~8 мест напросятся ПОСЛЕ stage 6** (миграция каскада стилей в reader,
   `ir-design.md §10` / `flow.ts:10-12`). Сегодня у этих ролей (резолвер шрифта рана,
   резолвер каскада, нумерация, bidi-анализатор) **ровно один потребитель — PDF-рендерер**.
   Выделять их в делегатов «ради переиспользования» сейчас — преждевременная абстракция.
   Их честная ценность сегодня — **дедуп внутри god-объекта** и подготовка шва, а не
   полиморфизм.

3. **~4 предложения — ловушка**: реестр писателей/ридеров и «канал потерь» (см. §6) —
   паттерны без второго потребителя, которые прячут асимметрию, а не устраняют её.

4. **Главная боль кодовой базы не в нехватке паттернов, а в одном god-объекте**:
   `src/pdf/styled-page-renderer.ts` — **3873 строки, 87 функций верхнего уровня**.
   Здесь GRASP «High Cohesion» и «Information Expert» нарушены сильнее всего. И именно
   здесь рез по уже существующему шву `layout↔emit` даёт максимум пользы при нулевом риске.

5. **Предусловие любого рефакторинга — автоматический байт-гейт** (см. §7). Сейчас его
   нет: существующие «byte-identical» тесты проверяют детерминизм *внутри одной ревизии*
   (`tagged-pdf.test.ts:378`, `pdfa.test.ts:193`, `facade.test.ts:51`), а перенумерацию
   объектов *между* ревизиями не ловит ничто. Без этого гейта «чистый рефакторинг»
   недоказуем.

## 1. Что уже хорошо: паттерны, реализованные без слова `class`

Кодовая база **уже паттерн-богата** — просто в функциональном стиле (интерфейс +
фабрика + замыкание вместо иерархий). Это сильная сторона, её надо сохранять.

| Паттерн (GoF/арх.) | Где | Evidence |
|---|---|---|
| **Strategy + Adapter** | `DocumentReader`/`DocumentWriter` и реализации | `adapters.ts:20-48`; `docx-reader.ts:115-139`, `xlsx-reader.ts:106-113`, `svg-writer.ts:60-65` |
| **Chain of Responsibility** | шрифтовая цепочка; sniff-детекция формата | `fonts/provider.ts:44-55` (первый byte-ответ выигрывает, `none` падает дальше); `facade.ts:68-69` |
| **Facade** | `createConverter().convert` | `facade.ts:65-122` |
| **Registry** | реестр ридеров с дефолтом/переопределением | `facade.ts:58-63` |
| **Flyweight / Identity Map** | `ResourceStore` (content-addressing, дедуп бесплатно) | `resources.ts:48-59` |
| **Value Object (branded)** | `Pt`, `ResourceId` — единицы не смешиваются на границах | `units.ts:12-17` |
| **Null Object** | `NO_FONT`; `EMPTY_STYLE_SHEET/NUMBERING/SECTION` | `provider.ts:35`; `docx-reader.ts:20-23,60-66` |
| **Extension Object / open-content** | `NativeBag` (raw-фрагмент по `formatId`), `Feature=string` | `ir/index.ts:31-33`; `features.ts:10` |
| **Memoization (closure-объект)** | `makeImageResolver`, `remoteFontProvider` | `docx-reader.ts:175-182`; `provider.ts:114-123` |

Вывод: **новые роли тоже должны быть интерфейс+фабрика, а не класс** — иначе ломается
tree-shaking (`package.json:33` `sideEffects:false`; класс с методами не тришейкается
по-методно). Классы оправданы только там, где они уже есть — изменяемое состояние
сборки: `PdfDocument`, `ResourceStore`, `OpcPackage`, `StructTreeBuilder`, `PathBuilder`.

## 3. Где делегирование реально напрашивается СЕЙЧАС ✅

Эти два — единственные с **живым вторым потребителем уже сегодня**. Делать в первую
очередь (особенно ДО заморозки PageDoc-схемы).

### 3.1. Изображения: роль `prepare` (декод/валидация) + роль `add` (эмиссия) — **P1**
*(канон A11, поглощает A10/A17; verdict: принять, byteRisk нет)*

- **Боль (реальная):** `embedImage` сейчас декодирует PNG **дважды** — на «пробе» в
  layout-фазе (`styled-page-renderer.ts:1498`, против одноразового `new PdfDocument()`
  ради skip-parity «битую картинку пропускаем») и снова на emit (`:1520`). Сам код
  помечает это TODO (`:1494-1496`).
- **Делегирование:** layout спрашивает у эксперта-по-изображениям только «декодируется
  ли? какие mime/размеры/alpha?» (роль `prepare`, чистая, без `doc.add`); emit отдаёт
  готовый результат в документ (роль `add`). Одноразовый `PdfDocument` исчезает.
- **Второй потребитель — настоящий:** `svg-writer.ts:198-202` (`sniffImageMime`) уже
  повторяет определение mime; после выноса он берёт mime/размеры из `prepare` вместо
  собственного сниффера. То есть роль реально нужна в двух адаптерах.
- **byteRisk: нет** — при дисциплине разреза «никаких `doc.add` в `prepare`» и сохранении
  порядка добавления (SMask раньше color-image, `image-xobject.ts:267→285`); `zlibSync`
  детерминирован. Под корпус-гейт по PNG+alpha.

### 3.2. Канонический «paint order» `PageItem` — один владелец для PDF и SVG — **P2**
*(канон A15; verdict: спорно→делать, byteRisk низкий)*

- **Боль (реальная):** порядок отрисовки `fills→images→borders→shapes→text`
  **продублирован независимо** в двух потребителях: `emitPageContent`
  (`styled-page-renderer.ts:3538/3564/3581…`) и `svg-writer.ts:74` (который прямо
  ссылается на PDF-emit). Оба обходят дискриминированный union `PageItem` своими
  `Array.filter` без проверки исчерпываемости — добавишь новый вид item'а, забудешь в
  одном из писателей.
- **Делегирование:** «план отрисовки страницы» (порядок + exhaustive-`switch` по
  `PageItem.type`) живёт в **одном** формат-агностичном месте; PDF и SVG предоставляют
  только «как нарисовать конкретный примитив». TS 5.5 даст exhaustiveness бесплатно.
- **Второй потребитель — настоящий:** оба writer'а, сегодня.
- **byteRisk: низкий** (`Array.filter` стабилен, порядок внутри групп сохраняется), но
  **строго до заморозки PageDoc** — потом это API.

## 4. Где делегирование напросится ПОСЛЕ stage 6 ⏳ (сегодня — дедуп внутри god-объекта)

Честная рамка: у этих ролей **сегодня один потребитель — рендерер**. «Второй потребитель»
(будущие `svg`/`docx`-writer, либо каскад-в-reader из stage 6) — пока гипотеза из
роадмапа. Поэтому сейчас это не «делегирование ради полиморфизма», а **(а) разрезание
god-объекта по швам и (б) устранение тройного пересчёта**. Польза реальная, но локальная.

### 4.1. Рез god-объекта по существующему шву `layout↔emit` — **P1**
*(канон A8; verdict: принять)*

Шов **уже физически есть**: `layoutStyledDocument` (`:526`) → `LaidOutDocument` →
`emitStyledPdf` (`:584`). Но он **течёт**: emit лезет в `options.*` вместо `laid.*`
(`emitStyledPdf` читает `options.resources` на `:1517`; `pdfaProfile` парсится в emit
дважды — `:1514`, `:1804` — вместо `laid.pdfaProfile`, построенного на `:540`).
**Действие:** герметизировать вход emit до `LaidOutDocument`, вынести две фазы в
`src/pdf/` отдельными файлами. byteRisk нет (перенос). Это разблокирует §3.

### 4.2. Кластер «эксперт не на месте»: FontResolver / StyleCascade / Numbering / Bidi
*(A14/A3, A12/A1, A2, A5 — все verdict «спорно»)*

| Роль | Боль сегодня | Второй потребитель | byteRisk | Когда |
|---|---|---|---|---|
| **FontResolver** (резолв рана→шрифт) | `runFontKeyAndParsed` гоняется в 2-3 фазах по одним ранам (collect `1692` → tokenize `1957/2010` → измерение таблиц `2622`); `measureSingleLine:2623` зовёт `resolveByStyle` напрямую, **игнорируя** `registriesByFamily`/`embeddedFonts` — скрытый баг | нет сегодня (`interface FontResource` даже не экспортирован, `:212`) | **средний** — нумерация `F1..Fn` зависит от порядка первого обращения (`1780-1791`) | дедуп можно сейчас (под гейт); делегат — после stage 6 |
| **StyleCascade-резолвер** (мемо `basedOn`) | тройной пересчёт каскада по одним ранам (call-sites `1691/1952/2009/2622`); `resolveMarker` зря берёт `numbering` каждый вызов (`state.ts:20`) | нет сегодня (только рендерер; «второй» = stage-6 reader) | нет (резолвер чист, `properties` readonly → identity-кэш безопасен) | parameter-object сейчас, переезд — stage 6 |
| **applyNumbering → core/numbering** | формат-нейтральный walk списков §17.9 заперт в PDF-слое (`1569-1638`) | нет сегодня | нет (`body.map(visit)` порядок идентичен) | stage 6 |
| **RunBidiAnalyzer → core/bidi** | коды RLE/LRE/PDF/OBJ (`1882-1886`) и параллельный `realIndexOfBidi` с `-1`-дырками (`2044-2077`) протекли в PDF-слой | гипотетически svg | низкий | по желанию |

**Рекомендация:** не делать big-bang. Сейчас — `parameter object` (`LayoutEnv`), который
станет домом для этих резолверов, и точечный дедуп с прогоном корпуса. Полноценное
делегирование в core — синхронно со stage 6, когда каскад реально переедет в reader и
у ролей появится второй живой потребитель.

### 4.3. Прочее по god-объекту (P3, делать последним — байт-чувствительно)
- **A6 ParagraphItemStream**: `wrap()` (`2250-2404`) — 155 строк, 4 параллельных массива
  (`2263-2271`), дубль магической `-10_000` (`2334` vs `knuth-plass.ts:42 FORCED_BREAK`,
  не экспортирована). Декомпозиция функции, не роль. byteRisk низкий.
- **A9 PageAssembler**: пагинация — 8 переменных состояния (`2977-2986`), хореография

  рефактор кластера (порядок push = разбивка страниц = байты). Только в самом конце.

## 5. Слоевые инверсии — объективный факт импортов ✅ (чинить, byteRisk нет)

Это не вкусовщина: направление зависимостей нарушено физически. Все подтверждены grep'ом.

### 5.1. Разорвать цикл `reader ↔ converter` + вынести Excel print-model — **P1**
*(канон C7, поглощает C11/C12/C13; verdict: принять)*

- **Двусторонний цикл:** `xlsx-reader.ts:22-27` импортирует 4 print-функции из
  `@/excel/xlsx-to-pdf`, а тот на `:55` импортирует `readXlsx` обратно.
- **Действие:** print-model (`worksheetToBody/sectionFromWorksheet/resolvePrintArea/
  resolvePrintTitleRows`) → отдельный модуль reader-слоя. Заодно мёртвый код:
  `mergeInfo` (`:186`), импорты `:43-52,58-61`.
- **Бонус:** эти функции **не публичны** (`excel/index.ts` их не реэкспортирует,
  `src/index.ts:89` отдаёт только `xlsxReader/readXlsx`) → compat-реэкспорт **не нужен**,
  вопреки опасениям. Перенос тривиален. byteRisk нет.

### 5.2. `po-helpers`/`xml-helpers`/`bytesInclude` → core — **P2**
*(канон C6, поглощает C2; verdict: принять)*

- `core` зависит от `word`: `chart-parser.ts:12,23` и `theme-parser.ts:10,11` тянут
  `@/word/po-helpers` (это **единственные** два нарушителя в core). `excel` зависит от
  `word`: `xlsx-reader.ts:21` → `bytesInclude` из `@/word/docx-reader`; `xlsx-reader.ts:11`
  → `EMPTY_STYLE_SHEET`.
- Файлы формат-агностичны (навигация по `preserveOrder`-дереву, ни одного OOXML-понятия).
  **Действие:** переместить в `core`, направление рёбер выправляется. byteRisk нет.

### 5.3. Единый разбор цветового узла DrawingML — **P2**
*(канон C1; verdict: принять)*

- **Дубль символ-в-символ:** `chart-parser.ts:161-172 colorMods` ≡ `drawing-parser.ts:412-423
  readColorMods`; `colorFromSolidFill` (`174-185`) ≈ `resolveColorNode` (`425-433`).
- **Микрорасхождение (важно при слиянии):** chart-версия возвращает на первом srgb/scheme
  с `val` даже при undefined-резолве (`:182`), word-версия продолжает обход (`437-439`).
  Консолидировать в `core/drawingml`, **выбрав одно поведение**, прогнать корпус.

## 6. Ловушки — делегирование/паттерн без второго потребителя ❌ (отклонено)

Здесь инстинкт «добавить паттерн» вреден — прячет асимметрию, не устраняет.

- **Реестр писателей / реестр пайплайнов** *(B1/B6/B8/B10 → отклонить)*: `opts.readers`
  имеет **ноль вызовов** во всём репо; третьего ридера не существует, роадмап
  (`ir-design.md §10`) кончается на stage 6. Реестр получит навсегда 2 записи. P1-«баг»
  фиктивного реестра (`facade.ts:113-116`) латентен — срабатывает только при
  использовании мёртвой @experimental-опции с несуществующим ридером. PDF-путь
  **структурно не может** идти через `DocumentWriter<PageDoc>`: конвертерам нужны сырые
  байты для авто-детекции шрифтов (`docx-to-pdf.ts:134`), а byte-identity требует legacy-
  маршрут (`facade.ts:2-5`). svg намеренно отдельный краш-тест (`svg-writer.ts:5`), а не
  сиблинг будущего pdfWriter. *(«ids зарезервированы в `adapters.ts:42`» — это пример в
  комментарии, не резервирование.)*
- **Канал потерь сквозь PDF-путь / LossSink** *(B3/B9 → отклонить)*: продюсеров потерь на
  PDF-пути **ноль** — `docxReader/xlsxReader` возвращают `losses:[]` хардкодом
  (`docx-reader.ts:112`, `xlsx-reader.ts:103`), рендер потерь не пишет. Канал понесёт
  пустой массив. Асимметрия контракта (`loss.ts:2-7`) реальна, но чинить нечего, пока
  никто не производит потери. Вернуться, когда reader/рендер реально начнут их писать.
- **A13 «сузить `LaidOutDocument`»** → отклонить: ломает **публичный** тип (`index.ts:121`)
  и `PageItem` (`:98`) без сегодняшней нужды; делать только синхронно с заморозкой PageDoc.
- **A16 LayoutEnv+CellPosition как самоцель** → отклонить как отдельную задачу: шумный
  механический diff в 3.8k-файле; делать только как побочный продукт §4.2 (env — дом для
  FontResolver).

## 7. Предусловие всего: автоматический байт-гейт — **P1, делать ПЕРВЫМ**
*(канон B11; verdict: спорно→обязательно)*

Главный инвариант проекта («чистый рефакторинг ⇒ PDF байт-в-байт») **не автоматизирован
между ревизиями**. Корпус-скрипт (`scripts/corpus`) дифферит к LibreOffice через
растеризацию — перенумерацию объектов он пропустит. Существующие «byte-identical» тесты
проверяют детерминизм *внутри одной ревизии* (`tagged-pdf.test.ts:378`, `pdfa.test.ts:193`,
`facade.test.ts:51` — facade-vs-direct). Cross-revision снимка нет.

**Действие:** хеш-манифест PDF-байтов на наборе фикстур (как `/tmp/baseline-hashes.txt`,
которым мы гейтили stage 1–6, но закреплённый в репо как тест). Это **предусловие** для
§3, §4, §5 — без него «byteRisk: нет» недоказуем.

## 8. Прочие проверенные находки (вне основных кластеров)

- **A4 единый `pickVariant` (fallback начертаний) — P2, принять.** Настоящий баг:
  `remoteFontProvider` (`provider.ts:126`) делает `fonts[v] ?? fonts.regular` — при
  best-effort CDN-наборе (`remote-fonts.ts:136-140`) bold-ран получит **regular**. Три
  реальных потребителя сегодня (`font-registry.ts:30-37`, `provider.ts:73-76`, `:126`) —
  редкий случай с живым ≥2 потребителем; free-функция тришейкается.
- **C4 ParseContext — P1, принять.** Пара резолверов `(resolveColor, resolveImage)` течёт
  через ~10 сигнатур (`parseDocument→parseBodyElements→tryExtractDrawing→parseTable→…→
  parseTableCell`). Цепочка уже **порвана**: `parseRun` зовёт `parseDrawing(child,
  defaultColorResolver)` (`document-parser.ts:382`) — теряет настоящий resolveColor. Context
  Object (литерал+readonly, не класс) оправдан. byteRisk нет (квирк сохраняется на время).
- **C5 per-part ImageResolver — P2, фикс-баг (medium).** `makeImageResolver` строится по
  rels **главной** части (`docx-reader.ts:166-183`, TODO на `:167-168`), но передаётся в
  `parseHeaderFooter` (`:240`) → картинки в колонтитулах резолвятся по чужому rId. Меняет
  выход осознанно (в правильную сторону) — вести как bugfix с пересогласованием golden.
- **C8 split `embedImage` — см. §3.1** (это и есть A11, кластеры пересеклись — одно дело).
- **B4 signPdf — P2, латентный correctness-баг.** Подпись ищется слепым сканом голого
  `/ByteRange` от начала файла (`signature.ts:157`); коллизия возможна, если вложение
  (`/AF` в PDF/A-3) содержит свой `/ByteRange…/Contents…`. `writer.build` уже ведёт
  офсеты (`writer.ts:51`). Ужесточить скан до фиксированного placeholder. byteRisk нет.
- **C3 OpcPackage rel-кэш — P3.** `getPartRelationships(MAIN_DOCUMENT_PART)` парсится
  заново 4× за конвертацию (`docx-reader.ts:170,189,211,231`); `REL_CORE_PROPERTIES`
  экспортирован, но мёртв (хардкод `'docProps/core.xml'`, `docx-reader.ts:36`). Приватный
  мемо-кэш безопасен.
- **C9 вектор-модель → core — P2.** `svg-writer.ts:22` тянет типы из `@/pdf/vector-graphics`
  (формат-агностичные, сам файл «owns nothing about OOXML»). НО `:21` тянет ещё и из
  `@/pdf/styled-page-renderer` — один перенос svg→pdf до конца не разрывает. Payoff неполный.
- **C10 axis-chrome хелпер — P3, хрупкий.** `buildFrame` (`chart-geometry.ts:138-319`) и
  `buildScatterScene` (`518-642`) дублируют ~75-90 строк хрома, но порядок `push` в
  буферы = порядок PDF-операторов (scatter: `576-629`). Любой наивный хелпер переставит
  операторы → расхождение. Только при побайтовом сохранении интерливинга.
- **B2 единый проектор FlowDoc→опции — P2.** Тройное дублирование проекции реально
  (`facade.ts:96-107`, `docx-to-pdf.ts:68-81`, `xlsx-to-pdf.ts:176-183`) и уже разъехалось
  (`sections` безусловно в `docx-to-pdf.ts:72` vs `length>0` в `facade.ts:100`). Вынести
  чистую функцию — DRY при добавлении поля FlowDoc. byteRisk низкий, но ключ-presence
  семантика обязательна к буквальному переносу под гейт.
- **B5 общий sync-помощник `convertOoxmlToPdf` — P2.** Конвертеры — близнецы (валидация
  fonts, двоение signature, embedSource, merge info). Схлопнуть дубль **двух** (не ждать
  гипотетического pptx). Многосемейный шрифтовой путь docx (`docx-to-pdf.ts:144-151`)
  остаётся в обёртке docx.

## 10. Кодекс ООП-изации reamkit (правила, которым подчиняется всё выше)

1. **Интерфейс + фабричная функция, не класс.** Класс — только для изменяемого состояния
   сборки (`PdfDocument`, `ResourceStore`, `OpcPackage`). Иначе ломается tree-shaking
   (`sideEffects:false`).
2. **Делегирование вводим только при ≥2 живых потребителях.** Один потребитель + «вдруг
   понадобится» = преждевременная абстракция. Будущий потребитель из роадмапа — повод
   подготовить шов (parameter object), а не вводить полиморфизм.
3. **`switch` по дискриминированному union идиоматичнее Visitor** (TS 5.5 inferred type
   predicates + exhaustiveness). Полиморфные объекты — для расширяемых СТОРОННИМИ точек
   (reader/writer), а не для закрытого набора `PageItem`.
4. **Любой «чистый рефакторинг» доказывается байт-гейтом** (§7). Нет гейта — нет права
   говорить «byteRisk: нет».
5. **Направление зависимостей: `format → core`, `converter → reader`, никогда обратно.**
   (§5 — текущие нарушения.)
6. **Публичные экспорты и subpath-ы (`.`, `./document-model`) — неприкосновенны** даже под
   @experimental.
